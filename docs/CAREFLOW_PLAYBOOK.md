# CareFlow Playbook
> 이 문서는 CareFlow의 모든 설계 결정과 그 이유를 기록합니다.
> 코드는 결과만 보여주지만, 이 문서는 맥락을 보여줍니다.

---

## 1. 정체성

**한 문장:** 간호학이 바라보는 방식으로, 일상 속 자신을 스스로 관찰하는 자기돌봄 도구.

**Health Awareness (건강 인식) = 신체 건강 + 정신 건강의 통합**

이론적 토대: Orem Self-Care Deficit Theory — Supportive-Educative System
(사용자가 스스로 돌볼 수 있도록 지식과 관점을 제공하는 역할)

간호학의 핵심: 사람을 **"생물심리사회적 통합체"**로 봄
- 신체 증상(이명, 어지럼증) ≠ 정신 증상(우울, 불안)이 아니라
- **모두 인간의 건강 상태를 반영하는 신호**로 통합적으로 봄
- CareFlow는 신체와 정신을 분리하지 않고 **함께 관찰하고 지원**

---

### 자료 수집: 객관적(O) vs 주관적(S)

**주관적 자료(S)**: 사용자의 직접 진술
- "귀에서 울려요", "불안해요", "밥맛이 없어요"
- **특징**: 개인의 경험과 느낌 (매우 중요하지만 혼자로는 충분하지 않음)

**객관적 자료(O)**: 관찰, 측정, 센서 데이터
- Apple Watch 센서: 심박수, 수면 패턴, 활동량
- 행동 관찰: 반응 속도, 활동 수준, 사회적 상호작용
- **특징**: 신뢰할 수 있는 측정과 관찰

**간호 판단**: O + S를 **함께** 분석해서 도출

## 2. 경계 원칙 — CareFlow는 무엇이 아닌가

### ⚠️ 핵심 원칙: CareFlow는 의학적 진단을 하지 않습니다

**CareFlow는 그 질병에 대한 인간의 반응(심리적 영향, 불안감, 생활 제약)에만 집중합니다.**

**신체 증상과 정신 증상을 동등하게 다룸:**
- 신체 증상(이명, 어지럼증, 비문증) ← 의료 진단 필요 + 심리적 지원
- 정신 증상(우울증, 불안장애, 공황장애) ← 정신과 진단 필요 + 심리적 지원
- 둘 다 **협력문제(Collaborative Problems)**로 분류
- 둘 다 **의료 전문가 연결 + 심리적 지원**의 원칙 적용

모든 신체/정신 증상에 대해:
- ❌ 진단을 하지 않음 (의료 전문가의 영역)
- ❌ 치료법을 제시하지 않음
- ❌ 약물 조언을 하지 않음
- ✅ 그 증상으로 인해 어떻게 느끼고 있는지 함께 관찰
- ✅ 일상 속에서 그 증상과 함께 살아가는 방법을 제안
- ✅ 필요 시 전문가 상담을 권함

| 아닌 것 | 이유 |
|---|---|
| 의료 진단 도구 | **CareFlow는 진단하지 않습니다.** 진단은 의료 전문가의 영역입니다. |
| AI 간호사 | 간호사의 전문적 판단을 임의로 학습시켜 대체하는 건 윤리적으로 위험 |
| 임상 의사결정 | 진료 결정은 면허가 있는 전문가의 영역. 우리는 자기관찰을 돕는 것 |
| 심리 상담 앱 | 심리학 접근이 아닌 간호학적 시선이 차별점 |
| 의학 정보 제공 서비스 | "우울증입니다", "이명입니다" 같은 진단명 제시 금지 |

**사용자가 느껴야 하는 건: "진단받았다"가 아니라 "나를 더 잘 알게 됐다"**

---

### 예시: 진단(v4.0) vs. 현상 파악(v4.1 "Related To")

**v4.0 표현 (❌ 진단처럼 들림 - 사용 금지)**
```
사용자: "매일 귀에서 윙윙거리는 소리가 들려요"
❌ CareFlow: "이건 이명입니다. 스트레스가 원인입니다."
```

**v4.1 표현 (✅ 현상 파악 - 관찰과 지원)**
```
사용자: "매일 귀에서 윙윙거리는 소리가 들려요"

자료 분석:
  S(주관): "귀에서 울려요", "밤에 더 신경 쓰여"
  O(객관): Apple Watch 데이터 - 야간 심박수 증가, 수면 단편화

✅ CareFlow 응답:
"귀에서 들리는 소리와 관련된 불안과 수면 방해를 경험하고 계신 거군요.

데이터를 보니:
- 밤 심박수가 평소보다 높아요
- 수면 중 깨는 횟수가 증가했어요

이런 현상은 먼저 이비인후과에서 전문가 상담을 받는 게 좋을 것 같아요.
그 후 우리는 그 소리와 함께 일상을 어떻게 관리할지 함께 살펴볼 수 있어요:
- 음향 마스킹 (백색소음, 자연음)
- 주의 전환 방법 (명상, 책 읽기)
- 수면 위생 개선"
```

### ICD-10 | KCD 의료 코드: 참고용일 뿐

문서 내에 ICD-10 | KCD 의료 코드가 있으면(예: R42 | H81 어지럼증),
이는 **CareFlow 개발팀이 증상의 의료적 배경을 정확히 이해하기 위한 참고용**입니다.
사용자에게는 절대 의료 코드나 진단명을 노출하지 않습니다.

---

### 🚦 NANDA 4가지 유형 × 신호등 색 (v4.1 "Related To" 프레임워크)

CareFlow는 **NANDA 간호학 전통**을 따르며 모든 사용자 상태를 **4가지 유형으로 분류**합니다.
이 분류는 CareFlow가 "의학적 진단을 하지 않고 현상 파악과 심리적 반응에 집중"하는 원칙을 구현하는 방식입니다.

| 🚦 색 | 유형 | NANDA 표현 | 자료 | CareFlow 응답 | 예시 |
|------|------|-----------|------|-------------|------|
| 🟢 | 안녕 간호진단 (Wellness) | Readiness for enhanced ~~ | S만 OK | 자기돌봄 강화 제안 | "Readiness for enhanced stress management" |
| 🟡 | 위험 간호진단 (Risk) | Risk for ~~ | O+S 필수 | 패턴 인식 + 예방 모니터링 | "Risk for anxiety related to sleep deprivation" |
| 🟠 | 실제 간호진단 (Actual) | Actual ~~ Related To | O+S 필수 | 현상 기반 심리적 지원 | "Disturbed sleep pattern related to anxiety" |
| 🔴 | **협력문제 (Collaborative)** | **Potential Complication of ~~** | **O+S 의료신호** | **즉시 의료 연결 + 심리 지원** | **"PC: Hearing loss — 이비인후과 상담"** |

#### 🔑 핵심: "협력문제 = Potential Complication (합병증)"은 CareFlow의 경계입니다

**협력문제(Collaborative Problems)**
= **Potential Complication (의료적 합병증 위험)**
= CareFlow가 의료 전문가에게 **반드시 연결**하는 지점

**합병증(Complication)의 의미**
- 기저 질환이나 증상으로부터 발생할 수 있는 **추가적 의료 위험**
- 예: 이명 → 청력 손상, 자살 위험 | 어지럼증 → 낙상/골절 | 우울증 → 자살 시도

| 🚦 | 기저 증상 | Potential Complication (합병증 위험) | CareFlow 역할 |
|----|---------|-----------------------------------|------------|
| 🔴 | 이명 | PC: Hearing loss / 이명성 자살 위험 | 의료 연결 + 청력 모니터링 + 심리 지원 |
| 🔴 | 어지럼증 | PC: Fall injury / 낙상·골절 위험 | 의료 연결 + 안전 가이드 + 심리 지원 |
| 🔴 | 우울증 | PC: **Suicide attempt** | **🔴 응급 연결 (1393)** |
| 🔴 | 불안장애 | PC: Panic disorder progression | 의료 연결 + 점진적 노출 심리 지원 |
| 🔴 | 공황장애 | PC: Agoraphobia development | 의료 연결 + 회피 행동 방지 심리 지원 |

```
❌ CareFlow가 하지 않는 것 (의료 전문가 영역)
   - 진단하기: "이건 이명입니다"
   - 합병증 판단: "청력이 손상되고 있습니다"
   - 치료하기: "이 약을 드세요"
   - 원인 단정: "스트레스가 원인입니다"

✅ CareFlow가 하는 것 (협력문제 = 합병증 관리)
   - 신호 감지: "귀에서 소리가 들린다는 게 맞나요? 얼마나 신경 쓰이세요?"
   - 의료 연결: "이비인후과에서 청력을 확인받는 게 좋을 것 같아요 (합병증 예방)"
   - 합병증 모니터링: "청력이나 마음의 변화가 있으면 바로 알려주세요"
   - 심리 지원: "그 증상으로 인한 불안을 함께 살펴봐요"
   - 안전 계획: "낙상 위험을 줄이기 위해 천천히 움직이기, 지팡이 사용하기"
```

## 3. CareFlow 4축 프레임워크 + NANDA 간호 진단 통합

v4부터 **NANDA-I 간호 진단 구조를 완전 통합** — 이론적 근거와 임상 신뢰도를 함께 확보.
Orem 이론 + Gordon Functional Health Patterns + NANDA 간호 진단을 참고했으나 라이선스 문제 없이 독자 설계로 운영.

### 축 구조

```
Body (몸) — 신체 건강 + 정신 건강 신호의 통합 관찰
  ├─ Sleep        잠과 회복 (신체/정신 모두에 영향)
  ├─ Energy       에너지와 활동 (신체/정신 상태 반영)
  └─ BodySignals  몸과 마음의 목소리
                  ← 심리 앱과의 차별점: 신체 신호를 관찰 대상으로 봄
                  ← 신체증상(이명, 어지럼증) + 정신증상(불안, 우울)을 통합 관찰

Emotion (감정)
  ├─ Tension      불안과 긴장
  ├─ EmotionFlow  감정의 흐름
  └─ SelfView     나를 대하는 방식

Connection (관계)
  └─ Relationship 연결과 관계

Meaning (의미)
  ├─ Direction    의미와 방향감
  └─ Growth       성장 준비

+ Safety Layer (위기 감지) — 4축과 별개로 모든 메시지를 교차 검사
  🔴 critical : 자해/자살 키워드 → 1393 즉시 안내, AI 호출 없음
  🟠 urgent   : 환청/환각/극심한 기능저하 → 1577-0199
  🟡 monitor  : 만성 패턴 → 정신건강복지센터 권유 + 자기돌봄 응답 병행
```

### 복수 도메인 동시 감지

"너무 지쳤어요, 불안하고 이유를 모르겠어요" 같은 메시지는
Energy(2점) + Tension(1점) + EmotionFlow(1점)이 동시에 감지됨.
AI는 단일 진단이 아닌 상위 3개 도메인을 종합해서 응답.

### 가중치(pt) 부여 기준

| 가중치 | 의미 | 예시 |
|---|---|---|
| 3 | 해당 도메인의 직접적·명시적 신호 | "잠을 못" → Sleep:3 |
| 2 | 해당 도메인의 주요 신호 | "지쳤어" → Energy:2, "불안" → Tension:2 |
| 1 | 이차적·간접적 신호 (다른 도메인의 부산물) | "지쳤어" → EmotionFlow:1 (번아웃이 감정에도 영향) |

**원칙:**
- 키워드가 그 도메인을 직접 서술하면 2-3점
- 해당 도메인이 *결과적으로* 영향받을 가능성이 높으면 1점 (cross-domain 신호)
- 점수 합산 후 가장 높은 도메인이 `primaryDomain`, 전체가 `activeDomains`로 AI에 전달됨
- 코드 위치: `nursingLogic.ts` → `DOMAIN_SIGNALS` 객체

## 4. 언어 사용 원칙 (NANDA 기반 v4)

### 써도 되는 말
- 자기돌봄, 자기관찰, 간호학적 시선
- 몸의 신호, 감정의 흐름, 연결, 의미
- "함께 들여다봐요", "스스로 알아가는 과정"
- "오늘의 나", "관찰 영역"
- NANDA 간호 진단: "수면 패턴 장애", "활동 불내증", "절망감" 등 (내부 가이드용)
  - 단, 사용자에게는 의학 진단명처럼 제시하지 말 것
  - 대신 "자극, 경로, 반응, 정서" 등으로 설명

### 쓰면 안 되는 말
- 간호 AI, AI 간호사, 24시간 곁의 간호사
- NANDA 코드 (NR-1.1 같은 라벨), NIC 코드 (사용자 노출 시)
- 우울증, 공황장애 등 의학 진단명
- "분석해요" (→ "함께 들여다봐요"로)
- "모든 게 잘 될 거예요" 류의 근거 없는 낙관

### NANDA 간호 진단을 사용자 언어로 번역하는 원칙

| NANDA 진단명 | 사용자에게는 이렇게 | 설명 |
|---|---|---|
| Sleep Pattern Disturbance | "자꾸 깨거나 잠을 못 자는 패턴" | 의학명이 아닌 현상 기술 |
| Activity Intolerance | "활동할 기력이 없는 상태" | 증상의 현실적 기술 |
| Hopelessness | "아무것도 나아질 것 같지 않은 느낌" | 감정과 경험을 직접 언급 |
| Social Isolation | "사람들과 연결되기 어려운 상태" | 관계적 어려움 기술 |

### 원칙
- **내부**: NANDA 진단으로 정확한 임상 근거 제시
- **외부(사용자)**: 감정, 신체 신호, 경험으로 번역하여 제시
- **목표**: 의학적 정확도와 사용자 친화성의 균형

### API 흐름 (NANDA 4가지 유형 분류 포함)

```
사용자 메시지 입력
  ↓
assessMessage()              # 사용자 상태를 NANDA 4가지 유형으로 분류
  ↓
NANDADiagnosisType 판단:

1️⃣ wellness       → 자기돌봄 강화 제안만 제공 (AI 호출)
2️⃣ risk          → 위험 신호 감지 + 예방 조언 (AI 호출)
3️⃣ actual        → 현재 심리적 어려움 지원 (AI 호출)
4️⃣ collaborative → ⚠️ 의료 전문가 연결 우선 (AI 호출 전 의료 안내)
  ↓
crisisLevel 동시 확인:
  critical/urgent → getCrisisResponse() 즉시 반환 (AI 호출 없음)
  monitor         → AI 응답 + 의료/심리 센터 정보 병합
  none            → 정상 응답
  ↓
buildSystemPrompt(assessment)  # NANDA 유형 + 4축 도메인 기반 지침
  ↓
AI 호출 (mock / openai / claude)
```

#### NANDA 협력문제(Collaborative) 처리 로직

```
협력문제 감지 → 즉시 의료 안내
  ├─ 신체증상: "이비인후과/안과 상담을 먼저 권해요"
  ├─ 정신건강: "정신건강위기상담전화 1577-0199"
  └─ 자살/자해: "자살예방상담전화 1393 (응급!)"
  ↓
+ 동시에 심리적 지원 제공
  ├─ "그 증상으로 인한 불안을 함께 살펴봐요"
  └─ CareFlow 자기돌봄 제안 병행
```

### AI 모드 전환

`.env.local`에서 `AI_MODE=mock|openai|claude`로 설정.
기본값 `mock` — API 키 없이 전체 기능 테스트 가능.

## 6. 의사결정 로그

### NANDA-I 프레임워크 제거 (v2 → v3)
- 이유: NANDA International은 비영리지만 상업적 사용에 라이선스 계약 필요
- 코드 내 타입명으로 숨겨도 법적으로 안전하지 않음
- 대안: CareFlow 독자 4축 프레임워크 (Orem + Gordon 참고, 복제 아님)

### "AI 간호사" 포지셔닝 포기
- 이유: 간호사의 전문성을 임의로 학습시켜 앱으로 만드는 건 윤리적 문제
- 서비스 경계가 불명확한 서비스는 사용자에게 위험
- 대안: Orem의 Supportive-Educative System 기반 자기돌봄 도구

### 복수 도메인 동시 감지 (단일 진단 → multi-domain)
- 이유: 실제 사람의 어려움은 단일 진단으로 떨어지지 않음
- "지쳤어요 + 불안해요"는 Energy + Tension 두 축이 동시에 작동
- AI에게 더 풍부한 맥락 전달 → 더 정확한 자기돌봄 제안 가능

### BodySignals 도메인 추가 (신체 신호 관찰 영역)
- 이유: 심리 앱들이 다루지 않는 영역. 간호학의 고유한 시선
- 심장이 빨리 뛴다, 목이 조인다, 밥맛이 없다 — 모두 자기관찰 정보
- **v4 확장**: 이명, 비문증, 어지럼증 등 만성 감각 증상도 포함
  - 이명(ICD-10: H93.1 | KCD: H93.1): 귀에서 울리는 소리 + 이로 인한 심리적 영향 추적
  - 비문증(ICD-10: H43.39 | KCD: H43.39): 눈 앞 검은 점 + 시력 불안 추적
  - 어지럼증(ICD-10: R42 | KCD: R42, ICD-10: H81 | KCD: H81): 회전감각 또는 현기증 + 균형 불안 추적
  - 메니에르병(ICD-10: H81.0 | KCD: H81.0): 이명+어지러움+난청 복합 증상 추적
  - 원칙: **의료 진단이 아닌 "증상으로 인한 심리적 영향"만 다룸**

## 8. Figma 활용 가이드

### 왜 Figma인가

CareFlow는 감정적으로 예민한 상황에 있는 사람과 대화하는 서비스예요.
색, 여백, 폰트 크기 하나가 사용자 경험을 완전히 바꿀 수 있어요.
Figma는 코드 없이 그 결정을 먼저 할 수 있는 공간이에요.

### CareFlow에 Figma가 구체적으로 도움이 되는 지점

**① 디자인 시스템 수립**
현재 Tailwind로 accent(초록), surface(연회색) 등을 쓰고 있는데,
이걸 Figma의 Design Token으로 정의해두면:
- 색상 변경 시 모든 컴포넌트에 자동 반영
- "위기 상태(🔴)"일 때 UI가 어떻게 달라져야 하는지 시각적으로 설계 가능
- Tailwind 설정값과 1:1 매핑해서 코드-디자인 불일치 방지

**② 채팅 UX 프로토타이핑**
지금 채팅 화면은 코딩으로 만들어졌는데, 앞으로 추가될 기능들
(관찰 영역 태그 표시, 다중 도메인 시각화, 히스토리 등)은
Figma에서 먼저 흐름을 잡고 코딩하는 게 훨씬 빠름.

**③ 감정적 톤 컨트롤**
CareFlow의 정체성("판단 없는 관찰 공간")은 시각 언어로도 표현되어야 해요.
- 어떤 폰트가 "임상적"으로 보이지 않는가
- 초록 계열 중 어떤 톤이 "차갑지 않은가"
- 위기 응답 시 빨간색을 쓸 것인가, 아니면 다른 방식으로 표현할 것인가
이런 결정을 Figma에서 A/B로 만들어보고 고를 수 있어요.

**④ 4축 프레임워크 시각화**
관찰 도메인(잠과 회복, 에너지와 활동 등)을 사용자가 어떻게 볼지
Figma에서 카드형, 그래프형, 태그형 등 여러 방식을 비교해볼 수 있어요.
코딩 없이 빠르게 판단 가능.

**⑤ 투자자/자문 피칭용 목업**
현재 있는 careflow_mockup.html보다 훨씬 완성도 높은 목업을
Figma로 만들어두면 피칭 자료에 바로 쓸 수 있어요.

### 당장 시작할 Figma 작업 순서

```
1단계  컬러/타이포 토큰 정의
       accent #28C840, surface #FFF8EC, primary-dark #FFFBF3
       폰트: 시스템 폰트 or Pretendard

2단계  컴포넌트 라이브러리
       MessageBubble (user / assistant / crisis 3종)
       QuickPrompt 버튼, TypingIndicator

3단계  채팅 화면 완성형 목업
       일반 대화 상태 / monitor 상태 / critical 상태 3개 프레임

4단계  관찰 영역 시각화 실험
       4축을 어떻게 사용자에게 보여줄지 (보여줄 경우)
```

### Figma 무료 플랜으로 충분한가

솔로 작업이면 무료 플랜(Starter)으로 충분해요.
파일 3개 제한이 있는데, CareFlow 1개 파일로 모든 컴포넌트와 화면을 관리할 수 있어요.
팀원이 생기거나 피칭 빈도가 높아지면 Professional 플랜($15/월) 고려.

## 9. 신체증상 진단 가이드 (v4 신규 추가)

### 이명(Tinnitus) — ICD-10: H93.1 | KCD: H93.1

**정의**: 귀에서 외부 음원 없이 울리는 소리가 들리는 증상

**CareFlow의 역할**
- ❌ 이명 자체의 의료 진단/치료
- ✅ 이명으로 인한 불안, 수면 방해, 일상 제약에 대한 심리적 지원

**감지 키워드**: "이명", "귀에서 소리", "귀울림", "윙윙", "삐익", "울림이 심해"

**BodySignals 매핑**
```
primary: BodySignals (weight 3) — 직접적 신체 신호
secondary: Direction (weight 1) — 이명 만성화 → "이게 평생 계속되면?" 무의미감
secondary: Tension (weight 1-2) — 밤에 울림 증폭 → 불안 악화
```

**응답 전략**
1. 의료 확인 먼저: 이비인후과 상담 권유
2. 진료 중/후 심리 지원: 음향 마스킹, 주의 전환, 수면 위생 개선
3. 장기적: 증상과 함께 살아가기, 삶의 의미 재구성

**응답 예시**
```
진단: "이명(울림) 지각 불안"
자극: 내이 과민 + 스트레스 누적
경로: 청각신경 과활성화 + 이에 대한 주의 편향
반응: 밤에 울림 증폭 → 잠 못 잠 → 피로 누적
정서: "이게 평생 계속되면?" → 불안, 무의미감
치료: 이비인후과 상담, 음향 마스킹, 신체 이완
```

### 비문증(Floaters) — ICD-10: H43.39 | KCD: H43.39

**정의**: 눈 앞에 검은 점이나 띠가 떠보이는 증상

**CareFlow의 역할**
- ❌ 비문증 자체의 의료 진단/치료
- ✅ 비문증으로 인한 시력 불안, 공포감에 대한 심리적 지원

**감지 키워드**: "비문증", "눈앞에 뭔가", "눈에 뭔가 떠", "검은 점", "띠"

**BodySignals 매핑**
```
primary: BodySignals (weight 3) — 시각 신호
secondary: Tension (weight 1) — 시력 손상 공포 → 불안
```

**응답 전략**
1. 의료 확인 먼저: 안과 진료 권유 (대부분 무해, 확인이 중요)
2. 진료 후 불안 관리: "이건 무해한 신체 반응", 주의 전환 기술
3. 장기적: 불안에서 수용으로의 전환

### 어지럼증/현훈(Dizziness/Vertigo) — ICD-10: R42 | KCD: R42, ICD-10: H81 | KCD: H81

**정의**: 세상이 도는 것 같은 회전감(vertigo) 또는 현기증(dizziness)

**CareFlow의 역할**
- ❌ 어지럼증 원인 진단 (BPPV, 메니에르병, 신경계 질환 등)
- ✅ 어지럼증으로 인한 낙상 공포, 활동 제약, 불안에 대한 지원

**감지 키워드**: "어지럼", "어지러", "어지럼증", "현기증", "현훈", "세상이 돌", "메니에르"

**BodySignals 매핑**
```
primary: BodySignals (weight 3) — 전정기 신호 (회전감/균형)
secondary: Direction/Tension (weight 1) — 낙상 공포 or 의미감 상실
```

**응답 전략**
1. 의료 확인 먼저: 이비인후과 또는 신경과 상담
2. 진료 중 일상 관리: 천천히 움직이기, 갑자기 고개 돌리지 않기, 눈 고정하기
3. 불안 관리: 낙상 공포 다루기, 활동 수준 조정
4. 장기적: 만성 어지럼증과 함께 살아가기

### 메니에르병(Meniere's disease) — ICD-10: H81.0 | KCD: H81.0

**정의**: 이명 + 난청 + 어지럼증 + 귀 충만감의 복합 증상 (내이 질환)

**CareFlow의 역할**
- ❌ 메니에르병의 의료 진단/약물치료
- ✅ 만성 복합 증상으로 인한 심리적 영향: 불안, 무의미감, 사회적 고립, 반복되는 발작에 대한 공포

**감지 키워드**: "메니에르", "이명+어지러움", "귀충만", "반복되는 발작"

**BodySignals 매핑**
```
primary: BodySignals (weight 3) — 다중 신체 신호 (이명, 어지러움, 난청)
secondary: Direction (weight 2) — "이게 언제까지?" → 무의미감, 장기 불안
secondary: Tension (weight 2) — 발작 공포, 악화 불안
```

**응답 전략**
1. 의료 팀과의 협력: 이비인후과 정기 진료 강조
2. 증상 트리거 추적: 발작 패턴 기록 (응력, 수면, 카페인 등)
3. 심리적 지원: 만성 질환 수용, 삶의 의미 재구성
4. 발작 시: 신체 안정화, 불안 관리, 지지 메시지

**응답 예시**
```
진단: "메니에르병으로 인한 복합 불안 및 무의미감"
자극: 반복되는 발작, 치료 불확실성, 사회적 제약
경로: 내이 질환 + 신경계 과민화 + 심리적 예기 불안
반응: 발작 공포 → 외출 회피 → 고립 → 의미감 상실
정서: "언제 또 발작할까?", "이대로 살아야 하나?" → 불안, 절망감
치료: 의료진과 협력, 발작 추적, 삶의 질 개선 기술, 지지 연결
```

### 통합 가이드: 신체증상 응답 작성 원칙 (v4)

#### ⚠️ 신체증상 응답의 원칙

**CareFlow는 증상 자체를 진단하지 않습니다.**
다만 **그 증상으로 인한 심리적 영향(불안, 무의미감, 일상 제약)을 함께 관찰합니다.**

**틀**: NANDA 6단계 응답 구조 (내부 가이드용)

```
1️⃣ 관찰 영역: "[증상 + 심리적 영향]" (진단명 아님)
   예: "어지럼증으로 인한 불안과 활동 제약"
2️⃣ 자극: 신체 신호 + 스트레스 요인
3️⃣ 경로: 신경생리 메커니즘 + 심리 증폭 기전
4️⃣ 반응: 증상 자체 + 이로 인한 일상 제약
5️⃣ 정서: 불안, 공포, 무의미감 등 심리적 영향
6️⃣ 함께 관리하기:
   **① 의료 전문가 우선**: "[전문과] 상담을 권해요"
   **② 현재 상태**: 진료 중/후 일상 관리 기술
   **③ 장기**: 증상과 함께 살아가기, 삶의 의미 재구성
```

**필수 원칙**
- 모든 신체증상 응답은 **"이비인후과/안과/신경과 상담을 먼저"**로 시작
- 내가 제시하는 건 진단이 아니라 "심리적 영향에 대한 관찰"임을 명확히
- ICD-10 | KCD 의료 코드는 내부 참고용일 뿐, 사용자 응답에는 절대 노출 금지

**금지 사항 (진단처럼 들리는 표현)**
- ❌ "이건 심리 증상입니다" → ✅ "심리적 영향이 크군요"
- ❌ "약을 줄이세요" → ✅ "의사선생님과 상담하세요"
- ❌ "완치될 거예요" → ✅ "함께 관리해나갈 수 있어요"
- ❌ "정신질환입니다" → ✅ "만성적 신체증상으로 인한 어려움"
- ❌ "[증상명]입니다" → ✅ "[증상]으로 인한 심리적 영향을 함께 살펴봐요"
- ❌ "스트레스가 원인입니다" → ✅ "스트레스가 이 증상을 악화시킬 수 있어요"
